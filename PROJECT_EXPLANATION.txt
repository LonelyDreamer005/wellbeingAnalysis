SCREEN TIME AND MENTAL WELLNESS ANALYSIS
Comprehensive Project Explanation and Justification

Date: 2026-04-14
Project: wellbeingAnalysis

=================================================================
1. EXECUTIVE SUMMARY
=================================================================
This project studies the relationship between screen usage behavior and mental wellness indicators using a cleaned, engineered dataset and an interactive machine learning web application.

The implementation has two practical goals:
1) Build a trustworthy, analysis-ready dataset from raw lifestyle data.
2) Make predictive analysis accessible through a browser interface for non-technical users.

In short, the project is not only about model training. It is a full workflow from data preparation to user-facing prediction, with model quality transparency through R2 reporting.


=================================================================
2. PROBLEM STATEMENT AND PURPOSE
=================================================================
Modern digital life increases daily screen exposure, which may affect stress, sleep, productivity, and overall wellness. However, raw lifestyle datasets are often inconsistent and not immediately usable for modeling.

This project addresses that by:
- Structuring and refining behavioral data.
- Enabling quick prediction experiments on wellness-related targets.
- Showing model performance to avoid blind trust in predictions.

Why this matters:
- It supports evidence-driven discussion of digital habits.
- It provides a reusable foundation for wellness analytics and future recommendation systems.

4. DATASET DESIGN AND JUSTIFICATION
=================================================================
The refined dataset contains 400 rows and 16 columns.

Core feature groups:
- Demographic/proxy context: age, gender, occupation, work_mode
- Screen behavior: screen_time_hours, work_screen_hours, leisure_screen_hours
- Wellness and lifestyle outcomes: sleep_hours, sleep_quality_1_5, stress_level_0_10,
  productivity_0_100, exercise_minutes_per_week, social_hours_per_week,
  mental_wellness_index_0_100
- Validation/engineering columns: screen_time_check, screen_time_diff

Why this schema is appropriate:
- It combines behavior (screen use), context (work style), and outcomes (wellness metrics).
- It enables both descriptive analysis and supervised prediction.
- It includes consistency checks (screen_time_check/screen_time_diff), which improve reliability.


=================================================================
5. DATA ENGINEERING LOGIC AND JUSTIFICATION
=================================================================
The notebook and supporting process include standard data engineering stages:

1) Loading and inspection
- Uses pandas for CSV ingestion and schema profiling.
- Purpose: establish data quality baseline before transformations.

2) Cleaning and normalization
- Removal of irrelevant, duplicate, or malformed content.
- Purpose: reduce noise and improve analytical validity.

3) Categorical encoding
- Encodes non-numeric fields to numeric values for model compatibility.
- Purpose: allow sklearn regressors to consume all relevant features.

4) Feature consistency validation
- Compares total screen time vs. work + leisure split.
- Purpose: identify logical mismatches and preserve dataset integrity.

5) Export of refined dataset
- Produces a stable analysis-ready artifact used by the app.
- Purpose: reproducibility and separation of engineering from serving.

Why this is the correct approach:
- Machine learning quality depends heavily on data quality.
- A reproducible preprocessing pipeline minimizes hidden errors.
- Explicit validation fields create auditability for assumptions.


=================================================================
6. APPLICATION ARCHITECTURE
=================================================================
The web app in app.py is intentionally lightweight and transparent.

Key functions:
- load_data(): finds refined dataset from known candidate paths.
- available_numeric_columns(df): exposes only numeric fields for prediction.
- train_best_model(df, input_col, target_col): trains two regressors and chooses the better one by test R2.
- index() route: validates user inputs, performs prediction, and returns response data to template.

Why this architecture is justified:
- Simple single-route Flask design keeps deployment and debugging straightforward.
- Utility functions isolate responsibilities (I/O, schema filtering, modeling).
- Restricting targets to numeric columns prevents invalid model calls.
- Input and target separation checks prevent degenerate modeling cases.


=================================================================
7. MODELING STRATEGY AND JUSTIFICATION
=================================================================
Current strategy:
- Supervised regression with one user-selected input feature and one selected numeric target.
- Candidate models:
  - LinearRegression
  - RandomForestRegressor (200 trees, fixed random_state)
- Model selection metric:
  - R2 score on held-out test split (80/20)

Why these choices are reasonable:
- LinearRegression provides a strong baseline and interpretability.
- RandomForestRegressor captures nonlinear behavior and interactions better than a simple line.
- Test-split evaluation avoids training-score illusion.
- Exposing model name and R2 to users promotes responsible interpretation.

Important interpretation note:
- Lower or negative R2 for some targets is possible and expected when one feature is insufficient.
- This does not imply application failure; it indicates limited explanatory power of the chosen input.

9. ERROR HANDLING AND ROBUSTNESS
=================================================================
Implemented safeguards include:
- File existence checks for dataset path.
- Numeric-column-only modeling options.
- Validation for invalid column selections.
- Rejection of identical input/target columns.
- Safe parsing of user numeric input with clear feedback.

Why these safeguards matter:
- They prevent common runtime failures.
- They protect model calls from invalid data shape/configuration.
- They improve reliability for interactive usage.


=================================================================
10. CURRENT LIMITATIONS (INTENTIONAL OR KNOWN)
=================================================================
1) Single-feature prediction by default
- Limits explanatory power for complex wellness outcomes.

2) Training occurs on each prediction request
- Easy to maintain, but less efficient than pre-trained model serving.

3) Correlational output
- Results show association patterns, not causal proof.

4) No confidence intervals
- Prediction uncertainty is not yet quantified in the UI.
